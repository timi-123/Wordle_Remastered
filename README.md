# Wordle Remastered

Десктоп игра **Wordle** изработена во **C# / WPF (.NET)**.

_Изработено од: да се договориме за редослед хахаха_

---

## 1. Опис на апликацијата

Апликацијата што ја развивме е модерна верзија на класичната игра **Wordle**, играта за погодување зборови на _The New York Times_ (оригиналната игра https://www.nytimes.com/games/wordle/index.html).

Целта на играта е да се погоди скриен **збор составен од пет букви** во најмногу **6 обиди**. По секој обид, секоја буква се обојува за да даде повратна информација (зелено / жолто / сиво) за дали истата била точно погодена, врз основа на која играчот го стеснува кругот од можни решенија.

За да ја направиме играта побогата и поразновидна, наместо само едно ниво ја проширивме со **три режими на играње**:

| Режим | Како се пристапува | Што нуди |
|-------|----------------|----------|
| **Unlimited Wordle** | копче `Daily Wordle` | Бесконечно играње Wordle со **статистика на сесијата** (одиграни, добиени, изгубени, % победи и распределба на победите по број на потребни обиди). |
| **Singleplayer (Timed)** | копче `Singleplayer Mode` | Игра во трка со времето, целта е да се погодат **што повеќе зборови** пред да истече избраното време. |
| **2 Player Battle** | копче `2 Player Mode` | Битка меѓу двајца играчи на иста машина, HP-битка во која точните букви му **нанесуваат штета** на противникот. |

Покрај чистиот и едноставен дизајн, имплементиравме реална проверка на зборови преку онлајн речник.

## 2. Упатство за користење

### 2.1 Главно мени

При стартување на апликацијата се отвора главното мени со три копчиња: **Daily Wordle**, **Singleplayer Mode** и **2 Player Mode**.

Со клик на копче се менува активниот поглед (преку `MainWindow.ChangeView(...)`), а копчето **← Back to Menu** во секој режим враќа назад во менито.

### 2.2 Unlimited Wordle

Наједноставниот, класичен режим. Се генерира случаен скриен збор и играчот го погодува во максимум од 6 погодоци. По завршување (победа или пораз) се прикажува порака и копче **PLAY AGAIN** за нова рунда, а десно се ажурира панелот со **статистика на сесијата**.

### 2.3 Singleplayer (Timed)

Прво се избира временско ограничување (**1 / 2 / 3 минути**).

Потоа започнува трка со време, целта е да се погодат **што повеќе зборови** пред да истече часовникот. Штом ќе се погоди зборот (или ќе се потрошат сите 6 обиди), екранот автоматски се ресетира со нов случаен збор. Системот за поени доделува:

- **+5 поени** за секоја нова **зелена** буква,
- **+2 поени** за секоја нова **жолта** буква,
- **бонус за погоден збор** во зависност од редот: `60 / 50 / 40 / 30 / 20 / 10` (побрз погодок = повеќе поени).

Кога времето ќе истече, се прикажува прозорец со конечниот резултат.

### 2.4 2 Player Battle

Режим за двајца играчи. Текот е:

1. **Време по обид**, се избира временски лимит по потег (10 / 20 / 30 секунди).
2. **Избор на зборови**, Играч 1 тајно внесува **5 збора со 5 букви** за противникот (секој се валидира преку речник), потоа екран за **предавање на уредот**, па Играч 2 ги внесува своите 5 збора.
3. **Битка**, играчите **наизменично** имаат еден обид за да го погодат зборот. Активен е само неговиот дел, а тастатурата на другиот е оневозможена.

Засновано е на база на **HP нокаут**, секоја точна буква му **одзема HP** на противникот (побрз обид = поголема штета), а погодувањето на цел збор е голем удар. Точните букви истовремено и **лекуваат** (HP се зголемува, но не над максимумот), а **истекувањето на времето** одзема HP. Првиот играч што ќе падне на **0 HP** губи.

### 2.5 Правила (боење)

По секој валиден обид, секоја буква се обојува според следново:

- **Зелено**, буквата е точна и на точна позиција.
- **Жолто**, буквата постои во зборот, но на друга позиција.
- **Сиво**, буквата воопшто не е во зборот.

Истите бои се пресликуваат и на тастатурата, за играчот лесно да следи кои букви ги пробал.

## 3. Претставување на решението

### 3.1 Архитектура

Проектот е WPF десктоп апликација организирана по **MVVM** (Model–View–ViewModel). Секој режим на игра има свој **View** (`UserControl`) и **ViewModel**, а навигацијата се одвива со замена на содржината на еден `ContentControl` во `MainWindow`. Сите погледи се обвиткани во `Viewbox` со фиксно платно, така што секогаш се видливи без рачно менување на големината.

| Датотека | Улога |
|---------------|-------|
| `MainWindow` | Главен прозорец, навигација (`ChangeView`). |
| `MainMenuView` | Главно мени. |
| `SinglePlayerView` + `SinglePlayerViewModel` | Тајмиран режим со поени. |
| `DailyWordleView` + `UnlimitedWordleViewModel` | Бесконечен режим со статистика. |
| `TwoPlayerView` + `TwoPlayerViewModel` + `PlayerBattleState` | Дуел за двајца. |
| `Controls/WordleBoardControl` | Повеќекратно употреблив grid + тастатура. |
| `Controls/TimerAndScoreControl` | Приказ на тајмер и поени. |
| `Models/LetterCell`, `Models/KeyModel` | Податочни модели. |
| `Services/WordValidator` | Проверка на зборови преку речник. |
| `ViewModels/ViewModelBase`, `ViewModels/RelayCommand` | MVVM инфраструктура. |
| `GameOverWindow` | Модален прозорец за крај на игра. |

### 3.2 Податочни структури

Состојбата на играта се чува во неколку клучни структури, кои се поврзани со UI преку `data binding` (секоја промена автоматски се рефлектира на екранот, бидејќи моделите имплементираат `INotifyPropertyChanged`):

**Една ќелија и едно копче** се претставени со мали модели:

```csharp
public class LetterCell : ViewModelBase   // едно поле од плочата
{
    public string Letter { get; set; }          // буквата во полето
    public Brush BackgroundColor { get; set; }  // боја (зелена/жолта/сива/празна)
}

public class KeyModel : ViewModelBase     // едно копче од тастатурата
{
    public string Key { get; set; }             // "A".."Z", "ENTER", "⌫"
    public Brush BackgroundColor { get; set; }  // боја на копчето
}
```

**Плочата и тастатурата** се чуваат како вгнездени `ObservableCollection` колекции (автоматски ги известуваат `ItemsControl` контролите за промени):

```csharp
// 6 реда × 5 полиња од grid
public ObservableCollection<ObservableCollection<LetterCell>> Guesses;

// 3 реда копчиња од тастатурата
public ObservableCollection<ObservableCollection<KeyModel>> KeyboardRows;

// сите копчиња во една листа (за брзо ажурирање на бои)
public ObservableCollection<KeyModel> KeyboardKeys;
```

**Скриениот збор** се избира случајно од листа од 80 зборови со 5 букви:

```csharp
private static string[] wordList = { "HELLO", "WORLD", "BRAIN", /* ... */ };
private string targetWord;  
```

За **режимот со двајца играчи**, состојбата на секој играч е во посебна класа `PlayerBattleState`, која ги содржи неговиот grid и тастатура, HP и неговите зборови:

```csharp
public class PlayerBattleState : ViewModelBase
{
    public string Name;
    public int HP;        
    public int MaxHP;
    public bool IsMyTurn; 
    public IReadOnlyList<string> TargetWords;  
    public int CurrentWordIndex;               
}
```

`TwoPlayerViewModel` ја оркестрира битката, чува два `PlayerBattleState`, наизменично менува чиј ред е, користи `DispatcherTimer` за одбројување и `Stopwatch` за прецизен бонус според брзината.

### 3.3 Валидација на зборови

Сите обиди и зададени зборови се проверуваат преку статичкиот сервис `WordValidator`, кој праќа барање до бесплатниот речник https://dictionaryapi.dev. Доколку зборот не постои, обидот се одбива. Ако нема интернет, методот враќа `true` за да не го блокира играчот:

```csharp
public static async Task<bool> IsValidWordAsync(string word)
{
    try
    {
        var response = await _httpClient.GetAsync(
            $"https://api.dictionaryapi.dev/api/v2/entries/en/{word.ToLower()}");
        return response.IsSuccessStatusCode;   
    }
    catch (HttpRequestException)
    {
        return true;   
    }
}
```

### 3.4 GUI

Grid-от и тастатурата се прикажуваат преку една повеќекратно употреблива контрола,`WordleBoardControl`. Таа се поврзува со `Guesses`, `KeyboardRows`, `ErrorMessage` и `EnterLetterCommand` од соодветниот ViewModel, па истата контрола ги опслужува **сите три режими**.

## 4. Опис на функција од изворниот код

Срцето на играта е функцијата за **боење на обид**, `EvaluateGuess`. Таа изгледа едноставно, но коректно ја решава најделикатната ситуација во Wordle: **повторени букви**.

```csharp
private void EvaluateGuess(string guess)
{
    List<char> remainingLetters = targetWord.ToList();

    // 1-ва пресметка, означи ги ТОЧНИТЕ (зелени) и тргни
    for (int i = 0; i < 5; i++)
    {
        if (guess[i] == targetWord[i])
            remainingLetters.Remove(guess[i]);
    }

    // 2-ра пресметка, обои ги полињата
    for (int i = 0; i < 5; i++)
    {
        char guessedLetter = guess[i];
        LetterCell cell = Guesses[currentRow][i];
        Brush color;

        if (guessedLetter == targetWord[i])
        {
            color = Brushes.Green;                         // точна позиција
        }
        else if (remainingLetters.Contains(guessedLetter))
        {
            color = Brushes.Goldenrod;                     // постои, друга позиција
            remainingLetters.Remove(guessedLetter);        // искористи едно појавување
        }
        else
        {
            color = Brushes.DarkGray;                      // ја нема
        }

        cell.BackgroundColor = color;
        UpdateKeyboardColor(guessedLetter.ToString(), color);
    }
}
```

**Како работи и зошто двета минувања:**

Наивниот пристап „жолто ако буквата е некаде во зборот“ е неуспешен при повторени букви. На пример, ако скриениот збор е `ROBOT`, а играчот напише `OOOOO`, само едно `O` смее да биде зелено/жолто, а не сите пет.

За тоа служи листата `remainingLetters` како **залиха на достапни букви**:

1. **Прво минување**, ги наоѓа сите букви на **точна позиција** (зелени) и веднаш ги **тргнува од залихата**, бидејќи се веќе искористени.
2. **Второ минување**, за секоја необоена буква проверува дали **сè уште има** такво појавување во залихата. Ако има тогаш е жолто и тргнува едно од залихата, а ако нема тогаш е сиво.

Така бројот на жолти + зелени за дадена буква никогаш не го надминува бројот на нејзини појавувања во скриениот збор, точно како во оригиналниот Wordle. На крај, `UpdateKeyboardColor` ја пресликува бојата и на тастатурата, притоа почитувајќи приоритет (зелено > жолто > сиво) за веќе обоено копче да не се обои погрешно.

Истата логика со двојно минување повторно се користи и при бодувањето во `SinglePlayerViewModel`, и при пресметката на штета/лекување во `TwoPlayerViewModel.ResolveGuessAsync`.
